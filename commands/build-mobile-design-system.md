# Build Mobile Design System from Scratch

You are guiding the user through a six-step wizard to create a complete mobile design system. There is no existing codebase required — this generates everything fresh based on their platform and project goals.

At the end, the user will have:
- A complete design token file in the correct format for their platform
- All selected components — fully built, token-using, accessible, all states
- `docs/components/[name].md` for each component
- `CLAUDE.md` with persistent design system context

---

## Opening Message

Begin by saying:

> **Build Mobile Design System — From Scratch**
>

> Six steps. I'll ask for your preferences at each stage, then generate everything. Let's start.

## Step 2 — Generate Tokens from Scratch

Say: "Now I'll generate a complete token set and write it in the format your platform expects. Every component we generate will reference these tokens."

### Generate the token set

Based on project type, design style, and primary brand color, derive all token values. Use the same color generation logic as the web version:

**Primary palette**: 9 shades (50–900) derived from the brand color
**Secondary palette**: 5 shades (100, 300, 500, 700, 900) — complementary or analogous
**Neutral grays**: 9 shades (cool for minimal/enterprise, warm for bold)
**Semantic**: success (green), warning (amber), danger (red), info (blue/cyan) — 3 shades each

**Spacing scale** — 4pt/dp base:
- xs: 4 | sm: 8 | md: 16 | lg: 24 | xl: 32 | 2xl: 48 | 3xl: 64 | 4xl: 96

**Typography scale**:
- Font: system font (SF Pro for iOS, Roboto for Android, or ask if they have a brand font)
- Scale: xs(12), sm(14), base(16), lg(18), xl(20), 2xl(24), 3xl(30), 4xl(36), 5xl(48)
- Weights: normal(400), medium(500), semibold(600), bold(700)
- Line heights: tight(1.25), normal(1.5), relaxed(1.625)

**Border radii**:
- Minimal: sm(4), md(8), lg(12), xl(16), full(9999)
- Bold: sm(8), md(16), lg(24), xl(32), full(9999)
- Enterprise: sm(2), md(4), lg(8), full(9999)

**Ask naming preference**:

Use AskUserQuestion:
- question: "How would you like to name your tokens?"
- header: "Token naming"
- options:
  - Semantic (colors.primary, spacing.md, typography.fontSize.base)
  - Numeric scale (colors.blue500, spacing[4], typography.fontSize.lg)
  - Keep it simple (I can rename later)

## Step 4 — Generate All Components

Say: "Generating [N] components in [platform] format. Each one includes all variants, states, and accessibility attributes — and references only token values."

For each component in the confirmed list:

1. Generate a complete, ready-to-use component in the detected platform's syntax
2. Every color, spacing, size, and radius value must reference a token (no hardcoded values)
3. Include all interactive states: default, pressed, focused (where platform supports it), disabled
4. Full platform-specific accessibility:
   - React Native: `accessibilityLabel`, `accessibilityRole`, `accessibilityState`, min touch target 44pt
   - Flutter: `Semantics` wrapper with `button: true`, `ConstrainedBox` with minWidth/minHeight 48
   - SwiftUI: `.accessibilityLabel()`, `.accessibilityHint()`, `.frame(minWidth: 44, minHeight: 44)`
   - Kotlin Compose: `semantics { contentDescription = ... }`, `sizeIn(minWidth = 48.dp, minHeight = 48.dp)`
5. Save to the platform's standard component location

After each component:
```
✓ Button — lib/widgets/button.dart
✓ Input — lib/widgets/input.dart
...
```

## Step 5 — Document All Components

Say: "Generating documentation for each component."

For each component generated in Step 4, write `docs/components/[component-name].md`.

Each doc must include:
- Overview: what it is, when to use it, when not to
- Variants: table with each variant and use case
- States: table for default, pressed, disabled, loading
- API / Parameters: platform-appropriate parameter table (props, constructor params, etc.)
- Design Tokens Used: table of tokens and their role
- Accessibility: platform-specific accessibility notes (accessibilityLabel, Semantics, etc.)
- Touch Targets: minimum sizes enforced and how
- Do's and Don'ts: 3–4 each
- Usage Examples: 2–3 platform-appropriate code examples

After each doc:
```
✓ docs/components/button.md
✓ docs/components/input.md
...
```
